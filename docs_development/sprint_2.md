# ТЗ: Спринт 2. Оживление главной страницы и умного поиска

**Цель спринта:** Заменить фейковую выдачу (заглушки) на реальные данные из таблиц `ads` и `ads_img`. Разработать ядро поиска и фильтрации по динамическим атрибутам (`ads_item_param_value`), чтобы платформа начала выполнять свою главную функцию — искать и показывать товары.

---

## Блок 1. Рефакторинг эндпоинта `f_api_get_list_ads` (`api/ads.php`)

**Проблема:** `f_api_get_list_ads` генерирует 20 хардкоженных карточек. Функция `f_db_get_list_ads` уже существует и умеет делать полный SELECT — нужно просто её подключить.

**Задачи для бэкенда:**

1. **Базовый SQL-запрос:**
   Убрать фейковый цикл. Вызвать `f_db_get_list_ads()` с условием `AND delete_on = 0 AND publication_on = 1`.

2. **JOIN с изображениями:**
   В `f_db_get_list_ads` добавить `LEFT JOIN ads_img ON ads_img.item_id = ads._id` (выбирать одно фото — по флагу `is_main = 1` или первое по `upload_date`). Поле вернуть как `html_img_src`.

3. **Пагинация (Infinite Scroll):**
   JS-функция `f_ads_item_list_scroll_load` передаёт номер страницы. Сервер должен принимать параметр `page` (или `offset`/`limit`) и формировать `LIMIT ? OFFSET ?`. Без этого скролл будет возвращать одни и те же записи.

4. **Приоритет платных объявлений:**
   Добавить сортировку `ORDER BY is_top DESC, _create_date DESC` — проплаченные объявления всегда сверху.

5. **Формат ответа:**
   JS-функция `f_ads_item_line_make` ожидает поля: `html_img_src`, `title`, `html_price`, `html_city`, `html_date`, `html_favorite_on`, `html_link_ad`. Убедиться, что `f_db_get_list_ads` их возвращает (сейчас возвращает `html_price_min`, `html_city` — нужна проверка совместимости).

## Блок 2. Исправление критических багов в `api/ads.php`

**Задачи:**

1. **Несовпадение имени функции `get`:**
   В `$gl_api_func_json` зарегистрировано `"get" => "f_api_get_ads"`, а объявлена функция `f_api_ads_get`. Переименовать одно из двух — вызов `/api/ads/get` сейчас фатально падает.

2. **Баг `$is_admin` в `f_api_ads_delete`:**
   Переменная `$is_admin` используется в условии без объявления. Добавить: `$is_admin = f_user_get()['type'] == 'admin';` в начало функции. Также проверить логику: сейчас условие `$item_json['user_id'] != f_user_get()['_id'] && $is_admin` выглядит инвертированным (должно быть `&& !$is_admin`).

3. **`f_db_insert` закомментирован в `f_api_ads_save`:**
   Раскомментировать и протестировать создание нового объявления. Убедиться, что `_create_user_id` и `user_id` проставляются корректно.

## Блок 3. Глобальный текстовый и категорийный поиск

**Проблема:** Строка поиска и выбор категорий в UI сейчас не влияют на выдачу.

**Задачи для бэкенда:**

1. **Поиск по тексту:**
   Если в API приходит параметр `search_query`, в SQL динамически добавляется: `AND (ads.title LIKE ? OR ads.description LIKE ?)`.

2. **Фильтрация по дереву категорий:**
   При клике на категорию нельзя искать только по одному ID. Использовать `f_arr_child_tree` (объявлена в `page/ads_create.php`) для сбора всех дочерних ID и добавить `AND ads.category_type_id IN (...)`.

3. **Геолокация:**
   Внедрить фильтр по `city_type_id`, если он передан с формы (поле уже есть в таблице `ads`).

## Блок 4. Логика динамических фильтров (`ads_item_param_value`)

**Проблема:** Параметры типа «Цвет», «Объём двигателя» хранятся в отдельной таблице `ads_item_param_value`. Обычный `WHERE` не работает для нескольких фильтров одновременно.

**Задачи для бэкенда:**

1. **Построение сложного SQL-запроса:**
   Для каждой пары `param_id` + `value` добавлять подзапрос:
   `AND ads._id IN (SELECT item_id FROM ads_item_param_value WHERE param_id=? AND value=?)`.

2. **Обработка диапазонов (Range Filters):**
   Для числовых параметров (цена, год, пробег) API должно поддерживать операторы `>=` и `<=`.

3. **Оптимизация:**
   Убедиться, что столбцы `item_id` и `param_id` в `ads_item_param_value` проиндексированы.

## Блок 5. Сортировка, Статусы и Модерация

**Задачи для бэкенда:**

1. **Глобальные условия выборки:**
   В каждый запрос к `ads` добавить: `AND delete_on = 0 AND publication_on = 1`.
   Если есть поле `moderation_status` — добавить `AND moderation_status = 'approved'`.

2. **Многоуровневая сортировка:**
   Реализовать параметр `sort` в API:
   - `newest`: `ORDER BY _create_date DESC`
   - `price_asc` / `price_desc`: `ORDER BY price ASC/DESC`
   - Платные объявления (`is_top`) всегда имеют наивысший приоритет.

## Блок 6. Интеграция с фронтендом и Бесконечный скролл

**Текущее состояние JS:** `f_ads_item_list_scroll_load` и `f_ads_filter_param_get` уже реализованы в `template/script.php` и ждут реального ответа от бэкенда.

**Задачи:**

1. **Совместимость полей ответа:**
   `f_ads_item_line_make` заполняет карточку по полям `html_img_src`, `title`, `html_price`, `html_city`, `html_date`, `html_link_ad`. Убедиться, что бэкенд возвращает именно эти ключи (сейчас `f_db_get_list_ads` возвращает `html_price_min` вместо `html_price` — нужно согласовать).

2. **Динамическая боковая панель фильтров:**
   При выборе категории фронтенд запрашивает доступные параметры через `f_arr_child_tree_options`. Отрисовывать соответствующие инпуты: чекбоксы для брендов, слайдеры для цен.

3. **Связка с AJAX:**
   При изменении фильтра или скролле JS собирает все активные фильтры через `f_ads_filter_param_get()` и отправляет POST через `f_ajax()` на `/api/ads/get_list`.
