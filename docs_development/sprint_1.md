# ТЗ: Спринт 1. Восстановление ядра пользователей и миграции БД

**Цель спринта:** Настроить надежный механизм обновления структуры БД, починить сломанное API пользователей (избавиться от несуществующей таблицы `pdd_user`), восстановить недостающие функции ядра и связать бэкенд с пользовательским интерфейсом (регистрация, вход, профиль).

## Блок 1. Система автомиграций (Создание `f_db_init.php`)

**Проблема:** Структура БД часто меняется (добавляются таблицы для чатов, избранного, магазинов). Накатывать `.sql` файлы вручную чревато ошибками.
**Решение:** Написать функцию, которая проверяет, какие SQL-патчи уже были применены к базе, и выполняет новые.

**Задача:**
1. В корне БД создать служебную таблицу `_db_migrations`, которая будет хранить историю выполненных обновлений.
2. Создать файл `func/f_db_init.php`.
3. Подключить вызов `f_db_init()` в безопасном месте (например, создать защищенный роут `/api/dev/db_init` или вызывать при старте админки).

**Концепт кода `func/f_db_init.php`:**

```php
<?php
// func/f_db_init.php

function f_db_init() {
    global $gl_db; // Предполагается, что соединение с PDO или MySQLi уже установлено в f_db.php

    // 1. Создаем таблицу для трекинга миграций, если её нет
    $sql_setup = "
        CREATE TABLE IF NOT EXISTS `_db_migrations` (
            `id` INT AUTO_INCREMENT PRIMARY KEY,
            `migration_name` VARCHAR(255) NOT NULL UNIQUE,
            `executed_at` DATETIME DEFAULT CURRENT_TIMESTAMP
        ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
    ";
    f_db_query($sql_setup); // Используйте вашу обертку для запросов

    // 2. Массив всех необходимых изменений БД (Миграции)
    // Ключ - уникальное имя миграции, значение - SQL запрос
    $migrations = [
        '001_fix_user_table' => "
            ALTER TABLE `user` 
            ADD COLUMN IF NOT EXISTS `phone_verified` TINYINT(1) DEFAULT 0,
            ADD COLUMN IF NOT EXISTS `accepted_terms` TINYINT(1) DEFAULT 0;
        ",
        // Сюда в будущем добавим:
        // '002_create_chats_table' => "CREATE TABLE `chats` ...",
        // '003_create_favorites' => "CREATE TABLE `user_favorites` ..."
    ];

    $applied_count = 0;

    foreach ($migrations as $name => $sql) {
        // Проверяем, была ли уже применена эта миграция
        $check = f_db_get_row("SELECT id FROM `_db_migrations` WHERE `migration_name` = ?", [$name]);
        
        if (!$check) {
            try {
                // Выполняем SQL
                f_db_query($sql);
                // Записываем в лог
                f_db_insert('_db_migrations', ['migration_name' => $name]);
                $applied_count++;
            } catch (Exception $e) {
                // Логируем ошибку и прерываем выполнение, чтобы не сломать БД
                die("Ошибка при выполнении миграции {$name}: " . $e->getMessage());
            }
        }
    }

    return "Успешно применено миграций: " . $applied_count;
}
```

## Блок 2. Починка бэкенда API (Ремонт `api/user.php` и `func/`)

**Проблема:** В файле `api/user.php` код пытается читать данные из таблицы `pdd_user`, которой нет в вашей схеме `schema_db.sql`. Также вызывается утерянная функция `f_user_type_ru`. Это приводит к фатальным ошибкам `500 Internal Server Error`.

**Задачи для бэкенда:**
1. **Рефакторинг `api/user.php`:** * Найти все SQL-запросы внутри `api/user.php` и заменить `pdd_user` на `user`.
   * Убедиться, что метод `save` (регистрация/апдейт) корректно работает со столбцами из актуальной таблицы `user` (`email`, `name`, `gender`, `type`, `activation_on`).
2. **Восстановление `func/f_user.php` (или где вы храните хелперы):**
   * Написать потерянную функцию перевода типов пользователей для вывода на фронтенде или в письмах.

**Пример потерянной функции:**
```php
function f_user_type_ru($type_en) {
    $types = [
        'admin' => 'Администратор',
        'user' => 'Покупатель / Частное лицо',
        'b2b' => 'Бизнес-аккаунт (Магазин)',
        'moderator' => 'Модератор'
    ];
    return isset($types[$type_en]) ? $types[$type_en] : 'Неизвестный тип';
}
```

## Блок 3. Создание недостающих страниц фронтенда

**Проблема:** Согласно документации `pages_modules_architecture.md`, роуты есть, а самих файлов `page/auth/login_registration.php` и `page/user/user_settings.php` физически нет. Без них пользователь не может взаимодействовать с платформой.

**Задачи для фронтенда:**

1. **Создать файл `page/auth/login_registration.php`:**
   * **UI:** Два таба или разделенные блоки "Вход" и "Регистрация" (с оглядкой на дизайн 999.md).
   * **Поля для регистрации:** Email, Пароль, Повтор пароля, обязательный чекбокс "Согласен с Terms of Service" (критично для UK).
   * **Поля для входа:** Email, Пароль.
   * **JS Логика:** Подключить вашу кастомную функцию `f_form_get(jq_modal_form)` для сбора данных из полей с атрибутом `field_name`.
   * **AJAX:** Отправлять собранный JSON через `f_ajax()` на эндпоинт `/api/user/save` (для регистрации) или соответствующий эндпоинт логина.

2. **Создать файл `page/user/user_settings.php`:**
   * **Доступ:** Проверить, что в `route.php` для этого пути стоит `user_check === true`.
   * **UI:** Форма редактирования личных данных. Поля: Имя (`name`), Телефон (`phone`), выбор типа профиля.
   * **UI Загрузки:** Блок для загрузки аватара (интеграция с готовым модулем `/api/upload/`).
   * **Безопасность:** Блок смены пароля (Старый пароль, Новый пароль).
   * **JS Логика:** Использовать вашу функцию `f_form_set(form_json, set_json)` для подстановки текущих данных пользователя в форму при загрузке страницы.

1. **Проверка `route.php`:** Убедиться, что после успешной авторизации сессия (`ssid`) надежно сохраняется в куки/PHP-сессию и переменная среды `$GLOBALS['user']` (или аналогичная, которую использует `f_user_get()`) корректно инициализируется при каждом запросе.
