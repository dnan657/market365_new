# ТЗ: Спринт 3. Коммуникация и Удержание (Чаты и Избранное)

**Цель спринта:** Реализовать систему личных сообщений между покупателем и продавцом, а также функционал «Избранного». Эти модули критически важны для превращения сайта из «витрины» в живую платформу и обеспечения возвращаемости пользователей (retention).

---

## Блок 1. Расширение БД (через `f_db_init.php`)

**Проблема:** В текущей схеме `schema_db.sql` полностью отсутствуют таблицы для хранения переписки и закладок.

**Задача:** Добавить в массив миграций `f_db_init.php` следующие структуры:

1.  **Таблица `chat`:** Реестр диалогов.
    * `_id` (PK), `_create_date`.
    * `item_id` (FK к `ads`) — чтобы понимать, по какому объявлению идет общение.
    * `user_buyer` и `user_seller` (FK к `user`) — участники сделки.
2.  **Таблица `chat_message`:** Сами сообщения.
    * `_id`, `chat_id` (FK), `user_id` (ID отправителя), `message_text` (text), `is_read` (tinyint), `_create_date`.
3.  **Таблица `user_favorite`:** Список закладок.
    * `user_id`, `item_id` (Уникальный составной индекс, чтобы нельзя было добавить одно и то же дважды).

---

## Блок 2. Бэкенд API (`api/chat.php` и `api/favorite.php`)

**Задачи для `api/chat.php`:**
1.  **`get_list`**: Возвращает список всех активных диалогов пользователя. Нужно делать `JOIN` с таблицей `ads`, чтобы выводить миниатюру товара и его заголовок в списке чатов.
2.  **`get_history`**: Возвращает историю сообщений по конкретному `chat_id`. При вызове этого метода все сообщения, где получатель — текущий пользователь, должны помечаться как `is_read = 1`.
3.  **`send`**:
    * Если диалога еще нет (первое сообщение от покупателя) — создает запись в `chat`.
    * Добавляет запись в `chat_message`.
    * *Важно:* Интегрировать `f_email_send`, чтобы продавцу уходило уведомление: «У вас новое сообщение по объявлению [Название]».

**Задачи для `api/favorite.php`:**
1.  **`toggle`**: Универсальный метод. Если товара нет в избранном — добавляет, если есть — удаляет.
2.  **`get_list`**: Возвращает массив объектов объявлений, которые пользователь пометил сердечком.

---

## Блок 3. Фронтенд: Интерфейс сообщений и Списки

**Проблема:** Нужно создать удобный UI, похожий на современные мессенджеры, используя имеющийся Bootstrap.

**Задачи:**
1.  **Страница `page/user/chats.php`**:
    * Верстка списка диалогов (слева — список, справа — пустое место или заглушка «Выберите чат»).
    * Использование `f_ajax` для подгрузки списка при входе.
2.  **Страница `page/user/chat_view.php`**:
    * Окно переписки. Вверху — закрепленная карточка товара (мини-сайт объявления).
    * Форма отправки: поле `textarea` и кнопка.
    * **Real-time (Polling):** Поскольку WebSockets — это отдельный сложный этап, на данном MVP реализовать «длинные опросы» или простой `setInterval` (раз в 5-10 секунд), который вызывает `get_history` для обновления сообщений.
3.  **Страница `page/user/favorites.php`**:
    * Отрисовка сетки объявлений. Можно переиспользовать шаблоны с главной страницы, но с кнопкой «Удалить из избранного».

## Блок 4. Логика безопасности и UX

1.  **Проверка прав:** В `api/chat.php` строго проверять, что пользователь имеет право видеть сообщения конкретного `chat_id` (т.е. он либо покупатель, либо продавец в этом диалоге).
2.  **Индикация:** Выводить красный бейдж с количеством непрочитанных сообщений в хедере сайта (нужна функция `f_chat_unread_count`).
3.  **Кнопка на карточке:** На странице `page/ads/item.php` реализовать переключение состояния сердечка (Избранное) без перезагрузки страницы через `f_ajax`.

# ТЗ: Спринт 3. Коммуникация и Удержание (Чаты и Избранное)

**Цель спринта:** Реализовать систему личных сообщений (безопасный мессенджер внутри платформы) и функционал закладок. Это позволит покупателям не терять товары, а продавцам — получать реальные лиды.

---

## Блок 1. Новые миграции БД (через `f_db_init.php`)

Необходимо добавить следующие структуры в массив миграций вашего скрипта инициализации.

**Задачи:**
1.  **Таблица `chat`**: Связующее звено между двумя пользователями по конкретному товару.
    * `_id` (PK), `_create_date`.
    * `item_id` (FK к `ads`) — товар, по которому идет торг.
    * `user_buyer_id`, `user_seller_id` (FK к `user`).
2.  **Таблица `chat_message`**: Хранилище самой переписки.
    * `_id`, `chat_id` (FK), `user_sender_id` (кто отправил).
    * `message_text` (текст сообщения).
    * `is_read` (tinyint, по умолчанию 0).
    * `_create_date`.
3.  **Таблица `user_favorite`**:
    * `user_id`, `item_id`. 
    * *Важно:* Создать UNIQUE индекс на пару `(user_id, item_id)`, чтобы один юзер не мог добавить один и тот же товар в избранное дважды.

---

## Блок 2. Бэкенд API (`api/chat.php` и `api/favorite.php`)

### 2.1. Модуль `api/chat.php`
Должен содержать следующие методы (`query`):
* **`get_list`**: Получение списка всех чатов текущего пользователя. 
    * *Логика:* Делаем `JOIN` с `ads` и `ads_img`, чтобы в списке чатов сразу видеть фото товара и его цену.
* **`get_messages`**: Загрузка истории переписки для конкретного `chat_id`. 
    * *Логика:* При вызове этого метода автоматически проставлять `is_read = 1` для всех входящих сообщений в этом чате.
* **`send`**: Отправка сообщения.
    * *Логика:* Если это первое сообщение от покупателя, API должно сначала создать запись в таблице `chat`, а затем в `chat_message`.
    * *Нотификация:* Вызывать функцию `f_email_send` для уведомления продавца о новом сообщении.

### 2.2. Модуль `api/favorite.php`
* **`toggle`**: Если запись в `user_favorite` есть — удалить, если нет — создать. Это удобнее для фронтенда (одна кнопка-переключатель).
* **`get_list`**: Возвращает массив ID или полных объектов объявлений, помеченных «сердечком».

---

## Блок 3. Фронтенд и UI-компоненты

**3.1. Интерфейс мессенджера (`page/user/chats.php` и `view_chat.php`):**
* **Список чатов:** Слева — список диалогов (аватар собеседника, миниатюра товара, последнее сообщение).
* **Окно чата:** Справа — сама переписка. 
* **Header чата:** Обязательно выводить мини-карточку товара сверху диалога, чтобы пользователи всегда видели, о чем договариваются.
* **Механика обновления:** Т.к. WebSockets (Node.js) требуют отдельной настройки, на текущем этапе используем **Long Polling** или `setInterval` (раз в 5-10 секунд) в JS для проверки новых сообщений через `get_messages`.

**3.2. Страница «Избранное» (`page/user/favorites.php`):**
* Верстка аналогична главной странице, но выводит только «сохраненки».

---

## Блок 4. Интеграция в карточку товара (`page/ads/item.php`)

1.  **Кнопка «Написать»**: Должна проверять авторизацию. Если юзер не вошел — редирект на логин. Если вошел — открытие модального окна или переход в чат.
2.  **Виджет «Сердечко»**: Должен менять цвет (заливка красным) мгновенно при клике (AJAX) через функцию `f_favorite_toggle()`.
3.  **Бейдж в хедере**: В главном меню сайта рядом с иконкой сообщений вызывать небольшую функцию (через `f_ajax` при загрузке), которая показывает количество непрочитанных (`is_read = 0`) сообщений.

## Definition of Done (Критерии готовности Спринта 3)

* [ ] Таблицы чатов и избранного созданы в БД через `f_db_init`.
* [ ] Покупатель может отправить сообщение продавцу прямо со страницы объявления.
* [ ] Продавец видит новое сообщение в своем кабинете и может ответить.
* [ ] Сообщения помечаются как прочитанные при открытии диалога.
* [ ] Товар добавляется в избранное и отображается в соответствующем разделе профиля.
* [ ] В хедере корректно отображается счетчик непрочитанных сообщений.
